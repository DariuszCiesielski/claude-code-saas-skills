# Credentials Vault â€” Secure Per-Project Credentials

## When to Use

- **Auto-activate**: You need a Supabase URL, API key, DB password, token â€” BEFORE asking the user
- **Auto-activate**: Authentication error (401, 403, "invalid key", "unauthorized")
- **On demand**: User provides credentials and wants them saved
- **On demand**: "what credentials do I have", "show saved keys"

## File Locations

### Per-project (overrides shared)
```
~/.claude/projects/{project-path}/credentials.env
```

`{project-path}` = project path with `/` replaced by `-`, prefixed with `-`.

Example: `/Users/johndoe/projects/acme-app`
-> `~/.claude/projects/-Users-johndoe-projects-acme-app/credentials.env`

### Shared (user's default keys)
```
~/.claude/shared-credentials.env
```

Contains the user's personal keys (OpenAI, Resend, DataForSEO, Cloudinary, etc.) that serve as defaults for ALL projects. A project can override any key in its own `credentials.env`.

### Read Order (inheritance)

```
1. Project credentials.env    <- highest priority (client/project-specific keys)
2. shared-credentials.env     <- fallback (user's personal keys)
3. none                       <- ask the user
```

**Rule:** If a key exists in both files, the project's credentials.env ALWAYS wins. Shared is only a fallback.

## File Format

```bash
# credentials.env â€” DO NOT COMMIT, DO NOT COPY
# Generated by credentials-vault

# === Supabase ===
# owner:client:acme
SUPABASE_URL="https://xxxxx.supabase.co"
SUPABASE_ANON_KEY="eyJ..."
SUPABASE_SERVICE_ROLE_KEY="eyJ..."
SUPABASE_DB_PASSWORD="password-with-special-chars!@#"

# === API Keys ===
# owner:personal â€” overrides shared (e.g., different OpenAI key for this project)
OPENAI_API_KEY="sk-proj-..."
```

**Format rules:**
- Values ALWAYS in double quotes
- Escape: `"` -> `\"`, `\` -> `\\`, `$` -> `\$`
- `#` comments for sections, blank lines between sections

## Owner Tagging

Each key can have an `# owner:` comment above it:

| Tag | Meaning | Where to store |
|-----|---------|----------------|
| `# owner:personal` | Your personal key | `shared-credentials.env` (default) or project `credentials.env` (if overriding) |
| `# owner:client:<name>` | Client's key | ONLY project `credentials.env` â€” never in shared |

**Client keys** (e.g., a client's Supabase project, a client's ElevenLabs agent) NEVER go into shared-credentials.env. Shared is exclusively for your personal keys.

## Read Procedure â€” MANDATORY Before Asking for Credentials

1. Determine the project path (`pwd` -> project-path)
2. Check project credentials: `cat ~/.claude/projects/{project-path}/credentials.env 2>/dev/null`
3. **Key found in project** -> use it, DO NOT ask the user
4. **Key NOT found** -> check shared: `cat ~/.claude/shared-credentials.env 2>/dev/null`
5. **Key found in shared** -> use it, inform: "Using shared key [NAME] from shared vault."
6. **Key NOT found anywhere** -> ask the user -> save (write procedure)

### Where to Save a New Key?
- **Personal key** (OpenAI, Resend, DataForSEO) -> `shared-credentials.env` (available across all projects)
- **Client key** (project's Supabase, client's ElevenLabs agent) -> project's `credentials.env`
- **Not sure?** -> ask: "Is this your personal key (shared) or specific to this project?"

**CRITICAL**: Never display full values. Mask them: `sk-proj-***...abc`, `sbp_***...xyz`.

## Write Procedure

1. Read existing file (if any)
2. Key exists -> update it (DO NOT duplicate)
3. New key -> append in the appropriate section
4. Save file with comment header
5. `chmod 600 credentials.env`
6. Confirm: "Saved [KEY_NAME] to credentials vault."

**Post-save validation**: run `scripts/validate.sh` (see [scripts/validate.sh](scripts/validate.sh))

## Handling Special Characters â€” CRITICAL

| Context | Rule | Example |
|---------|------|---------|
| In `.env` file | Double quotes + escape `"` `\` `$` | `DB_PASSWORD="pa\$\$word!"` |
| In shell commands | ALWAYS use single quotes | `psql -W 'pa$$word!'` |
| In JS/TS code | No issues â€” string literal | `const pw = "pa$$word!"` |
| In `export` bash | Single quotes | `export DB_PASSWORD='pa$$word!'` |

When a user provides a password with `!$\`"#`, warn:
> "Password contains special characters. In shell I MUST use single quotes."

## Working Practices â€” What to Save in Project MEMORY.md

When a connection works with specific credentials:
```
- Supabase connection: works, DB password contains `!` â€” shell requires single quotes (credentials-vault)
```

**In MEMORY.md** save ONLY: that credentials work, special requirements, reference to `(credentials-vault)`.
**NEVER** save actual key values in MEMORY.md.

## Pitfalls

### 1. Shell interprets `!` in double quotes
```bash
# BAD: bash interprets ! as history expansion
echo "password123!"
# GOOD: single quotes
echo 'password123!'
```

### 2. `$` in password treated as variable
```bash
# BAD: $pecial -> empty variable -> "pass"
export PW="pass$pecial"
# GOOD: literal
export PW='pass$pecial'
```

### 3. Overwriting file instead of appending
NEVER overwrite with `>`. Read -> modify in memory -> save with Write/Edit tool.

### 4. Credentials in git history
If credentials ended up in the repo â€” deleting the file is not enough. Values remain in history. You must rotate the keys.

### 5. Missing chmod 600
Without `chmod 600` other users on the machine can read the file. ALWAYS set after creation.

### 6. Displaying full values
Never show the user full keys in output. Always mask them.

### 7. JWT in ANON_KEY expires
Supabase ANON_KEY is a JWT â€” it has an expiration date. If you get "JWT expired" despite a correct key, check the date in the payload (base64 decode the middle part).

### 8. Dotenv vs credentials.env
The credentials.env file is NOT automatically loaded by dotenv. The agent must read it explicitly. Do not mix with the project's `.env.local` â€” these are separate mechanisms.
